# Calibration test harness

End-to-end synthetic dataset for calibrating the multi-channel scoring formula. This is the development/QA tooling used to derive the empirical defaults shipped in [`stash-extract-scraper/config.py`](../../stash-extract-scraper/config.py) and the bridge `BRIDGE_FEATURIZE_*` env vars.

> Run-by-run findings live in [`CALIBRATION_RESULTS.md`](CALIBRATION_RESULTS.md). For the architectural invariants the calibration is testing against, see [`CLAUDE.md`](../../CLAUDE.md). For the user-facing runbook (no calibration, just operate the bridge), see [`docs/HOW_TO_USE.md`](../HOW_TO_USE.md).

---

## 1. Why this exists

### 1.1 The original problem

The bridge does fuzzy image search between two asymmetric corpora:

- **Domain (Stash side)**: each video has a cover image and a sprite sheet of ~81 frames sampled across runtime.
- **Codomain (extractor side)**: each metadata record has 0..N images that may correspond to any or no frames in the input video set.

The naïve approach — Hamming-distance every codomain image against every sprite frame, take the mean of `1 - d/hash_size²` per record — falls apart on real corpora because the underlying assumption (an "average similarity" is meaningful) is wrong. Specifically:

1. **N varies wildly across records.** Some records have 1 image, some have 100. A weighted average over a flat M×N pair grid lets a record with many borderline images outrank a record with one perfect match.
2. **Video duration varies.** A 30-second clip and a 30-minute clip both produce 81 sprite frames, so the fidelity of "this frame matches" depends on duration in ways the mean doesn't capture.
3. **Per-image signal quality varies.** A blank wall, a fade-to-black, or a credits screen produces a high-frequency hash that matches lots of unrelated content. These shouldn't carry the same weight as a content-rich frame.
4. **Distribution shape carries information.** Pair-similarities of `[1.0, 0.1, 0.1, 0.1]` mean "one image strongly matches"; `[0.325, 0.325, 0.325, 0.325]` means "all images weakly support the same video." Both sum to 1.3, but the second is broader corroborating evidence — and a mean treats them identically.

The architectural response — **multi-channel scoring** with sharpened evidence-union, per-image quality and uniqueness weighting, count saturation, and a distribution-shape term — is documented in [`CLAUDE.md`](../../CLAUDE.md) §13. **This directory is the empirical instrument that confirmed the architecture and tuned its parameters.**

### 1.2 The labeling problem

Multi-channel scoring needs sensible defaults for `image_gamma`, `image_count_k`, `image_uniqueness_alpha`, `image_min_contribution`, plus the per-channel uniqueness threshold and alpha. These are calibration parameters — there's no first-principles derivation, only empirical tuning against labeled `(scene, expected_record)` pairs. Acquiring such labels for a real Stash + extractor pair is expensive (manual review of every match) and license-encumbered (real adult content can't be checked into the repo).

### 1.3 The synthetic-dataset trick

Use the **same source videos** for both Stash-side scenes (full videos → sprite sheets generated by Stash) and extractor-side records (random sampled frames as record images). This produces ground-truth `scene_X ↔ record_X` pairs without manual labeling. Pexels (free CC-0 stock video) supplies a license-clean, varied corpus.

Negative controls — videos imported into the extractor's record set but **not** into Stash — measure false-positive rate. A perfect matcher returns empty for any Stash scene that would have matched these.

## 2. Pipeline overview

```
┌────────────────────────┐
│ 1. gen_dataset.py      │  Pulls videos from Pexels (or yt-dlp), samples
│    (Pexels default)    │  random frames per video, builds records.json
└──────────┬─────────────┘  + assets/ + ground_truth.json
           │
           ▼
┌────────────────────────┐
│ 2. mock_extractor.py   │  Serves the dataset via the bridge ↔ extractor
│    (FastAPI on :12001) │  contract. Drop-in replacement for the real
└──────────┬─────────────┘  site-extractor service.
           │
           ▼
┌────────────────────────┐
│ 3. import_to_stash.py  │  Copies videos to Stash's library, triggers
│                        │  metadataScan, creates the studio, links scenes.
└──────────┬─────────────┘  Skips negative-control videos.
           │
           ▼
┌────────────────────────┐
│ 4. Bridge against mock │  Point the bridge at the mock extractor;
│    extractor URL       │  featurization runs end-to-end on real data.
└──────────┬─────────────┘
           │
           ▼
   tests/integration/test_calibration.py walks ground_truth.json,
   queries each Stash scene, compares top-1 to expected_record_id,
   writes a JSONL run-log under tests/calibration/runs/.
```

## 4. Step 1 — Generate the dataset

```bash
python -m tests.calibration.gen_dataset \
    --source pexels \
    --target 50 \
    --dataset-dir tests/calibration/dataset \
    --job-name "Calibration Test Site"
```

What you get:

```
tests/calibration/
  dataset/
    schemas/
      video_scene_schema_<hex>.json
    jobs/
      calib_<hex>/
        job.json                      # {id, name, completed_at, extraction_config}
        records.json                  # list of records (Video Scene shape)
        assets/
          0000_cover.jpg              # frame thumbnails
          0000_img_0.jpg
          ...
    ground_truth.json                 # the calibration fixture
  .video_cache/                       # downloaded source videos (re-used across runs)
    pexels_<id>.mp4
    ...
```

**Sizing**: start with `--target 50` for a fast end-to-end smoke. Once the pipeline is verified, re-run with `--target 500` for the full corpus. The `.video_cache/` survives between runs, so re-runs only download new videos.

**For the alternate source**: `--source ytdlp`. Slower, but pulls broader variety. Requires `yt-dlp` on PATH.

## 6. Step 3 — Import videos into Stash

The Stash library path is whatever directory Stash is configured to scan. Pick a subdirectory inside it for the calibration set; `import_to_stash.py` will create it if needed.

```bash
python -m tests.calibration.import_to_stash \
    --dataset-dir tests/calibration/dataset \
    --video-cache-dir tests/calibration/.video_cache \
    --stash-target /path/to/stash/library/calibration \
    --symlink
```

`--symlink` saves disk by symlinking from `.video_cache/` to the Stash library. Drop the flag if Stash doesn't follow symlinks (depends on the deployment — Docker volume mounts often don't).

The script:
1. Copies/symlinks every non-negative-control video.
2. Triggers `metadataScan` and waits for completion (typically minutes per video for sprite generation).
3. Creates the studio matching `--job-name` from gen_dataset (`Calibration Test Site` by default).
4. Links each imported scene to that studio.

**Negative controls** are deliberately *not* imported — they exist in the extractor's records but have no matching Stash scene. The matcher should return empty for any Stash scene that would have matched these.

> Stash's sprite generation on 50 fresh videos can take 30+ minutes, and on 500 it's hours. Plan for an overnight run when scaling up.

## 7. Step 4 — Point the bridge at the mock extractor

Two options:

**Option A — Run a second bridge instance** alongside production, with different env:

```bash
docker run --rm -d --name stash-extract-db-calib \
  --network extractor_network \
  -p 13001:13000 \
  -e STASH_URL="$STASH_URL" \
  -e STASH_API_KEY="$STASH_API_KEY" \
  -e EXTRACTOR_URL=http://host.docker.internal:12001 \
  -e DATA_DIR=/data \
  -e BRIDGE_LIFECYCLE_ENABLED=true \
  -e BRIDGE_NEW_SCORING_ENABLED=true \
  -v "$(pwd)/data-calib:/data" \
  $(docker compose config --hash stash-extract-db | awk '{print $1}')   # or rebuild
```

**Option B — Re-point the existing bridge** by editing `.env`:

```bash
# In .env:
EXTRACTOR_URL=http://host.docker.internal:12001
BRIDGE_LIFECYCLE_ENABLED=true
BRIDGE_NEW_SCORING_ENABLED=true
```

```bash
docker compose up -d --force-recreate stash-extract-db
docker compose logs -f stash-extract-db | grep -E 'featurize|featurization'
```

Either way, the bridge will:
1. On startup, scan jobs from `EXTRACTOR_URL` (the mock) and enqueue featurization for the calibration job.
2. Featurize all records' images — fetching thumbnail bytes from the mock's `/api/asset/...` endpoint.
3. Compute baselines + uniqueness across the calibration record set.

Watch progress:

```bash
curl -s http://localhost:13001/api/featurization/status | jq
```

When `ready == 1` and `in_progress == 0`, you're ready to match.

## 8. Step 5 — Run a calibration sweep

The harness lives at [`tests/integration/test_calibration.py`](../../tests/integration/test_calibration.py). It walks `ground_truth.json`, queries each Stash scene against the bridge, and writes a JSONL run-log to `tests/calibration/runs/<timestamp>_<label>.jsonl`.

```bash
python -m tests.integration.test_calibration \
    --label "n500_baseline" \
    --bridge-url http://localhost:13001 \
    --debug
```

Per-cell parameter overrides:

```bash
for gamma in 2.0 2.5 3.0 3.5; do
  for k in 0.25 0.5 1.0 2.0; do
    python -m tests.integration.test_calibration \
      --gamma $gamma --count-k $k \
      --label "g${gamma}_k${k}"
  done
done
```

Each run reports `precision@1`, `mean_reciprocal_rank`, `mean_top1_score`, and `n_negatives_correctly_empty`. Compare across cells, identify the winner, append a "Run N" section to [`CALIBRATION_RESULTS.md`](CALIBRATION_RESULTS.md).

For an interactive ad-hoc query (no harness):

```bash
SCENE_ID=<stash_scene_id_from_ui>
curl -s -X POST 'http://localhost:13001/match/fragment?debug=1' \
  -H 'Content-Type: application/json' \
  -d '{
    "scene_id": "'"$SCENE_ID"'",
    "mode": "search",
    "image_mode": "both",
    "threshold": 0.001,
    "limit": 5,
    "hash_algorithm": "phash",
    "hash_size": 8,
    "sprite_sample_size": 8,
    "image_gamma": 3.5,
    "image_count_k": 0.25,
    "image_uniqueness_alpha": 1.0,
    "image_channels": ["phash", "color_hist", "tone"],
    "image_min_contribution": 0.05,
    "image_bonus_per_extra": 0.1
  }' | jq '.[] | {title:.Title, code:.Code, score:.match_score, image_S:._debug.image.composite}'
```

Look up the expected record in `ground_truth.json`:

```bash
jq -r '.[] | select(.video_basename | contains("pexels_<id>")) | .expected_record_id' \
  tests/calibration/dataset/ground_truth.json
```

## 9. Resetting between runs

```bash
# Wipe the dataset (regenerates on next gen_dataset run)
rm -rf tests/calibration/dataset

# Wipe the video cache too (forces re-download)
rm -rf tests/calibration/.video_cache

# Wipe the bridge's calibration features (forces re-featurization)
rm -rf data-calib/                                    # if using Option A
# OR clear the relevant rows from the bridge's SQLite cache:
docker exec stash-extract-db sqlite3 /data/stash-extract-db.db <<'SQL'
DELETE FROM job_feature_state WHERE job_id LIKE 'calib_%';
DELETE FROM corpus_stats WHERE job_id LIKE 'calib_%';
DELETE FROM image_uniqueness WHERE job_id LIKE 'calib_%';
DELETE FROM image_features
  WHERE source IN ('extractor_image', 'extractor_aggregate')
    AND ref_id LIKE 'calib_%';
DELETE FROM extractor_results WHERE job_id LIKE 'calib_%';
DELETE FROM extractor_jobs WHERE job_id LIKE 'calib_%';
SQL

# Wipe the imported Stash scenes + studio (manual via Stash UI is safer)
```

## 10. Notes from prior calibration cycles

This section preserves load-bearing context discovered during the multi-channel rollout that isn't otherwise visible from the code.

### 10.1 Asyncio event-loop blocking (fixed)

PIL/imagehash/numpy compute is synchronous and CPU-bound. Earlier versions of the bridge called these directly inside async coroutines; with 491 records × ~5 images × 3 channels ≈ 7,000 hash operations queued, the cumulative event-loop block was 5–15 minutes — during which the bridge could not respond to ANY HTTP request, including its own `/health` endpoint. Calibration runs hung at the first match request.

The fix wrapped CPU-bound compute in `asyncio.to_thread()` at five hot spots in `bridge/app/matching/image_match.py` and `bridge/app/matching/featurization.py`. The regression test `tests/unit/test_lifecycle.py::TestEventLoopResponsiveness::test_db_query_latency_stays_low_during_featurization` asserts max DB query latency < 150 ms during featurization (typically <30 ms in practice). See `CALIBRATION_RESULTS.md` Run 1b.

### 10.2 FK violation in `_gate_features_ready` (fixed)

On a never-seen job, the gate tried to insert into `job_feature_state` (which has FK to `extractor_jobs`) before the parent row existed. Fix calls `inv.ensure_job_results_fresh(j)` inside the gate before enqueuing. Captured by `tests/unit/test_lifecycle.py::TestWorker::test_enqueue_creates_state`. See `CALIBRATION_RESULTS.md` Run 1b.

### 10.3 The "magnet record" failure mode

At the original default of `image_gamma=2.0`, three records (`scene_0057`, `scene_0353`, `scene_0226`) accumulated wins via broad-shallow evidence-union — many low-quality, low-uniqueness sims summed to a misleading composite. Sharpening (γ=3.5) broke up most of them; the count-saturation rebalance (k=0.25) closed most of the rest. The remaining magnet (`scene_0057`) has 9 pHash images all with `c_i=1.0` because none of them are flagged as near-duplicates at the global pHash threshold of 0.85 — yet collectively they accumulate to a small but winning composite for a scene with no real match. **No global tuning closes it.** Per-channel uniqueness was implemented and validated, but did not close the gap on the Pexels corpus (Run 7).

### 10.4 Negative-control behavior

Search mode always returns top-N by design — there's no "no match" decision. A search-mode confidence floor (`IMAGE_SEARCH_FLOOR`) was implemented and validated; on the Pexels corpus the composite distributions of weak-but-correct positives and weak-but-incorrect negatives overlap, so no single floor value separates them without dropping legitimate positives. The mechanism ships disabled by default; users with sharper corpora may enable at 0.10–0.20 (see Run 6).

### 10.5 Side-car services for an isolated calibration run

When running calibration alongside a production bridge, two side-car processes are stood up:

- `mock-extractor` on `:12001`
- `calibration bridge` on `:13001`, env: `BRIDGE_LIFECYCLE_ENABLED=true`, `BRIDGE_NEW_SCORING_ENABLED=true`, `EXTRACTOR_URL=http://127.0.0.1:12001`, fresh `DATA_DIR=$(mktemp -d -t bridge-calib-XXXXXX)`

Stop them when finished:

```bash
for port in 13001 12001; do
  pid=$(lsof -i :$port -P -n -sTCP:LISTEN -t 2>/dev/null)
  [ -n "$pid" ] && kill $pid
done
```

The production bridge (`:13000`) and Stash (`:9999`) are not touched at the code level by calibration runs — only the mock extractor's data is involved.

## 11. Known limitations

- **Pexels API rate limits** (200 req/hr free tier) cap the corpus to roughly the number of distinct queries × 15 per hour. For >500 videos, batch across runs (the `.video_cache/` survives between runs).
- **Frame timing**: thumbnails are sampled in [5%, 95%] of runtime. If a video has near-uniform content from 10% to 90% (e.g., a static talking-head), the channel A/C signal is degenerate. This mirrors a real-world failure mode you'd want calibration to expose.
- **Negative controls are honest negatives only on the Stash side**. The records exist in the mock extractor's response with full image refs. A perfect matcher returns empty/low-score for the negative scenes; an over-eager one will surface them as false positives.
- **30-scene sample size**: the calibration runs documented in `CALIBRATION_RESULTS.md` use a fixed 30-scene sample (`random.seed=0`) of the 500-scene corpus for tractable wall-clock per cell (~12 minutes vs ~3 hours full). Wilson confidence at 25/26 = 96.2% is `[82%, 99%]`. Tuning above this corpus's resolution requires a larger sample.
- **Pexels-style mixed-content corpora silence channel C (tone)**. The global uniqueness threshold of 0.85 collapses tone `c_i` for typical natural-scene content; this is the intended behavior on this corpus (tone noise > tone signal at scale). Corpora with controlled lighting (monochrome film, surveillance footage) may benefit from a stricter per-channel tone threshold — the mechanism is shipped but unused by default. See Run 7.
